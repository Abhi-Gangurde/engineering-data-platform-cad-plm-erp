Section 01
Project Overview
Purpose, scope, source systems, and end-to-end architecture of the Engineering Data Platform.

1.1 Purpose

The Engineering Data Platform ingests, transforms, and serves data from three source domains — CAD metadata, PLM (Product Lifecycle Management), and ERP (Enterprise Resource Planning) — into a governed, query-optimised Delta Lake on Azure Data Lake Storage Gen2, exposed via Unity Catalog for analytics and applications.

Why this platform exists

Engineering organisations accumulate product and cost data in siloed systems (CAD tools, PLM databases, ERP cost tables). Without a unified data platform, analysts manually export CSVs, join them in Excel, and produce stale reports. This platform eliminates manual effort, enforces data quality, and delivers a single source of truth updated on every pipeline run.

1.2 Source Systems

CAD Metadata

CSV files exported from a CAD tool to an on-premises file server. Files are date-stamped (e.g. cad_metadata_2026-03-10.csv). Ingested via batch comparison against last-processed date.

PLM — Revisions

Azure SQL DB table plm_revisions. Contains part revision lifecycle with last_modified watermark. Loaded incrementally via timestamp-based CDC.

PLM — BOM

Azure SQL DB table plm_bom. Bill of Materials linking assemblies to component parts. Watermark column: last_modified.

ERP — Cost

Azure SQL DB table erp_cost. Supplier unit costs per part. Watermark column: last_updated (different column name from PLM tables).

1.3 Master Architecture

End-to-end platform architecture

1.4 Design Principles

  • Medallion architecture — Bronze (raw) → Silver (cleansed) → Gold (aggregated) → Serving (consumer-shaped). Each layer has a single responsibility and is independently replayable.
  • Incremental-first — every source table uses watermark-based CDC. Full reloads only happen on seed (first run with watermark = 1900-01-01) or explicit reset.
  • File-based control state — watermark JSON stored on ADLS instead of a separate SQL control table, eliminating an extra dependency and simplifying the architecture.
  • Delta UPSERT everywhere — all writes use merge-on-key semantics. Re-running any layer is idempotent — no duplicates, no manual cleanup.
  • Governance as first class — Unity Catalog tags, column masks, audit logging, and DQ metrics are built in from day one, not added later.

4.3 PL_PLM-ERP_Ingesion

Incremental CDC pipeline for all three SQL source tables. Reads watermarks from ADLS JSON, extracts changed rows, writes to staging Parquet, runs Delta UPSERT, and updates the watermark on success.

PL_PLM-ERP_Ingesion — activity flow
Why the watermark updates AFTER the Data Flows, not after the Copies

If the watermark were updated after copy_plm_bom but the subsequent df_upsert_plm_bom failed, the next run would skip those rows — they would never reach the Delta table. By depending on all three Data Flow activities succeeding, the watermark only advances when data is confirmed written to Delta. A failed run re-processes the same window on retry.

Section 05
Watermark & CDC Design
How incremental ingestion works, why a JSON file was chosen over a SQL control table, and how failure safety is guaranteed.

5.1 Watermark JSON structure

The watermark control file lives at bronze/plm/watermark/plm_watermark.json on ADLS Gen2.

[
  {
    "table_name":     "plm_revisions",
    "last_watermark": "2026-03-10 09:00:00",
    "watermark_col":  "last_modified",
    "merge_key":      ["part_id", "revision"],
    "adls_path":      "bronze/plm/revisions"
  },
  {
    "table_name":     "plm_bom",
    "last_watermark": "2026-03-10 10:05:00",
    "watermark_col":  "last_modified",
    "merge_key":      ["assembly_id", "part_id"],
    "adls_path":      "bronze/plm/bom"
  },
  {
    "table_name":     "erp_cost",
    "last_watermark": "2026-03-11 10:30:00",
    "watermark_col":  "last_updated",
    "merge_key":      ["part_id", "supplier_name"],
    "adls_path":      "bronze/erp/cost"
  }
]
Why JSON file instead of SQL control table

A SQL control table requires an additional Azure SQL instance or a dedicated table in the source database — adding a dependency that doesn't belong in the source system. The JSON approach stores control state on the same ADLS where data lives, uses ADF's native Lookup JSON source (no extra linked service), and is overwritten atomically via a single PUT REST call. For lightweight pipelines with a fixed set of tables, this is simpler, cheaper, and sufficient.

5.2 CDC extract query pattern

SELECT *, SYSUTCDATETIME() AS _ingestion_ts
FROM   dbo.plm_revisions
WHERE  last_modified > '@{activity('Lookup-GetLatestWatermark').output.value[0].last_watermark}'
  AND  last_modified <= SYSUTCDATETIME()

Upper-bound explained

The AND last_modified <= SYSUTCDATETIME() clause sets a hard upper bound at the moment the query runs. Without it, a row written to the source during pipeline execution (after the Lookup reads the old watermark but before the Copy finishes) would be skipped on the next run — because the watermark would be set to utcNow() which is after that row's timestamp. The upper bound prevents this race condition.

5.3 Failure safety guarantee

Failure scenarioWhat happensNext run behaviour
Copy activity failsData Flow and Web activity don't run (dependency: Succeeded)Watermark unchanged → same window re-processed → idempotent
Data Flow failsWeb activity doesn't runWatermark unchanged → re-runs the Copy AND Data Flow
Web activity failsData is in Delta, watermark not updatedNext run re-extracts same window → Delta UPSERT deduplicates → no duplicates in Delta

Section 06
Mapping Data Flows
Three ADF Data Flows that perform Delta UPSERT merge from staging Parquet into the Delta bronze tables.
Why Mapping Data Flow instead of Copy activity sink directly to Delta

ADF Copy activities can write to Delta, but only in append mode. They cannot perform MERGE operations. If a source row is updated (e.g. a cost changes from 1500 to 1650), a Copy-to-Delta would add a second row with the new value alongside the original. The Delta table would then contain two rows for the same part/supplier key — making it unusable for reporting. A Mapping Data Flow with AlterRow(upsertIf(true())) and key columns generates a MERGE statement that updates existing rows and inserts new ones, maintaining exactly one row per business key.

Section 07
Bronze Layer Notebook
nb_bronze_layer_ingestion — objective, functions, and table outputs.
bronze_plm 4 tables Raw / as-landed Delta MERGE

7.1 Objective

Read all data landed by ADF (Delta files for PLM/ERP, CSV files for CAD) from ADLS Gen2 and write it into Unity Catalog as Delta tables under bronze_plm schema. Add standard audit columns. Preserve all source data unchanged — no filtering, no transformation, no business logic.

Why keep Bronze as a raw copy

Bronze is the immutable record of what arrived from the source system. If a Silver transformation has a bug, you can re-run Silver from Bronze without re-running ADF. If a source system sends bad data, Bronze preserves it — allowing root cause analysis after the fact. Transforming in Bronze would destroy this audit trail.

7.2 Functions

add_audit_columns(df, source_path, domain)
Appends 4 standard audit columns to any DataFrame: _source_path (ADLS origin), _source_domain (cad/plm/erp), _bronze_ingested_at (pipeline run timestamp), _bronze_pipeline_run (run identifier). These columns allow any row in Bronze to be traced back to its exact pipeline run.
write_bronze_table(df, table_name, merge_keys, partition_cols)
Writes DataFrame to Unity Catalog bronze schema. First run: CREATE TABLE + full insert. Subsequent runs: Delta MERGE on merge_keys (upsert). Optional partition_cols for large tables. Prints row count after write.

part_cost_view (conformed join)

Three-way join of Silver tables producing the primary conformed dataset for Gold consumption:

plm_bom (silver)
  LEFT JOIN plm_revisions (silver, is_current_revision = true)  ON part_id
  LEFT JOIN erp_cost (silver)                                    ON part_id

→ Derived: total_assembly_cost = cost × quantity
→ Derived: cost_data_available = cost IS NOT NULL
→ Derived: revision_data_available = revision IS NOT NULL
Why produce a conformed view in Silver (not Gold)

Gold notebooks should focus on aggregation and business modelling, not joining. By producing the conformed part_cost_view in Silver, Gold reads one pre-joined table instead of performing three-way joins itself. This also means multiple Gold tables (assembly cost, supplier cost, etc.) all read from the same consistent join — no risk of different Gold tables using slightly different join logic.

10.1 Views vs Marts — design decision

Why both views AND materialised marts

Views are zero-cost to refresh (they always reflect the latest Gold data) but incur compute on every query. Use for interactive Power BI dashboards and ad-hoc exploration where freshness matters more than performance. Marts are materialised Delta tables — they cost one write per pipeline run but serve queries from storage (no compute join overhead). Use for scheduled reports, large Excel exports, and API backends where consistent query latency matters. The platform provides both so each consumer type gets the right trade-off.

  • VACUUM — 7-day retention (168 hours). Run on every governance execution. Removes files older than retention window to control storage cost. Why 7 days: allows 7 days of time-travel queries for debugging without excessive storage cost.
  • OPTIMIZE + Z-ORDER — compact small Parquet files and co-locate data by key columns. Run on every governance execution. Why every run: incremental writes produce many small files; Z-Order ensures point lookups on part_id remain fast.

Error Handling & Known Issues
Every error encountered during the build — root cause, fix, and prevention.
ErrorRoot causeFix appliedPrevention
Incorrect syntax near '2026' ADF expression with type: Expression strips single quotes from '@{...}' — the datetime value reached SQL without quotes, being interpreted as arithmetic. Changed to string interpolation mode using @concat('...''', value, '''...') where '' is an escaped literal quote. Always use @concat() for building SQL strings with dynamic values. Never embed @{} inside SQL strings in Expression mode.
PublicAccessNotPermitted Web activity sent the PUT request without a bearer token — Managed Identity was not configured (Authentication = None). Set Authentication = Managed Identity, Resource = https://storage.azure.com/. Assigned Storage Blob Data Contributor IAM role to ADF MSI. Always use MSI auth for ADLS. The Resource field must be exactly https://storage.azure.com/ — not the storage account URL.
The format of value 'application/jsonContent-Type: application/json' is invalid Two headers were merged into one value — Content-Type was set in both the Headers grid and accidentally concatenated. Ensured each header is its own key-value row in the ADF Web activity Headers section. Added required x-ms-blob-type: BlockBlob as separate row. Web activity headers must be set as individual key-value pairs — never concatenate header name and value.
Array index '1' cannot be selected from empty array Watermark JSON file was corrupted (malformed JSON from a failed Web Activity write) — Lookup returned an empty array []. Manually restored valid JSON in Azure Storage Explorer. Added validation activity to check array length and reseed if empty. Add ValidateWatermarkNotEmpty IfCondition activity after Lookup. Set Web activity retry = 1.
ForEach ran all 3 copies per item = 9 executions All 3 Copy activities were hardcoded inside the ForEach. On each iteration, all 3 ran regardless of which table the current item represented. Removed ForEach. Each Copy activity now directly references value[0], value[1], value[2] from Lookup output. All 3 run in parallel. Use ForEach only when the number of items is dynamic. For fixed tables, reference Lookup output by index directly.
Silver notebook paths pointed to bronze folder All 4 notebook parameters in PL_Master_Pipeline had adb/scripts/bronze/ as the path — Silver, Gold, Serving notebooks were in the wrong default path. Updated defaultValue paths: silver → scripts/silver/, gold → scripts/gold/, serving → scripts/serving/. Each notebook layer must be in its own Databricks folder matching the parameter path.
Section 15
Operations Runbook
How to handle common operational scenarios — adding tables, resetting watermarks, and re-running failed layers.

15.1 Add a new source table

  1. Add a new entry to plm_watermark.json on ADLS with last_watermark: "1900-01-01 00:00:00"
  2. Add the new entry to the Web Activity @concat() body in PL_PLM-ERP_Ingesion
  3. Add a Copy activity for the new table referencing value[N].last_watermark
  4. Create a new Mapping Data Flow with the correct merge keys
  5. Add the Data Flow activity to the pipeline, depending on the Copy
  6. Update the watermark Web activity to depend on the new Data Flow
  7. Add the new table to Bronze, Silver, and Gold notebooks

15.2 Reset a watermark (force full reload)

  1. Open bronze/plm/watermark/plm_watermark.json in Azure Storage Explorer
  2. Set last_watermark to "1900-01-01 00:00:00" for the target table
  3. Save the file
  4. Trigger PL_Master_Pipeline — the pipeline will re-extract all rows
  5. Delta UPSERT will update existing rows in place — no manual cleanup needed

15.3 Re-run a failed notebook layer

Every notebook is idempotent — re-running it produces the same result as the original run. Simply trigger PL_Master_Pipeline again. ADF will start from the beginning, but because ADF pipelines do not skip previously-succeeded activities within the same run, you may prefer to trigger the specific Databricks notebook directly from the Databricks workspace for faster recovery.

15.4 Interpret quarantine alerts

  1. Query govern_plm.alert_log for today's date where alert_type = 'Quarantine'
  2. Query the relevant quarantine table: silver_plm.{table_name}_quarantine
  3. Group by _rejection_reason to identify the most common failure
  4. Trace the source rows in bronze_plm.{table_name} using _bronze_ingested_at
  5. Fix the source data or adjust the validation threshold in the Silver notebook
  6. Re-run the Silver notebook — corrected rows will flow through on the next run
