03 — ADLS Gen2 — Medallion Architecture
🥉
BRONZE
Raw / Immutable
Raw data stored as-is from all three sources. No transformations applied. Audit columns added by ADF.
cad_metadata_raw plm_bom_cdc_raw plm_revision_cdc_raw erp_cost_incremental_raw
+ Metadata columns: ingestion_time source_system batch_id file_name
🥈
SILVER
Cleaned / Standardized
Databricks (PySpark) applies cleaning: null handling, deduplication, type casting, CDC merge logic, unit normalization, surrogate key generation.
parts_clean bom_clean revisions_clean cost_clean
Transformations: Null handling Deduplication Schema enforce CDC MERGE Incremental MERGE
🥇
GOLD
Star Schema
Business-ready Fact & Dimension tables. Star schema modeled for analytical queries. SCD Type 2 on revisions. Optimized with partitioning and Z-ordering.
dim_part dim_assembly dim_supplier dim_revision (SCD2) dim_date
fact_bom fact_cost fact_design
📊
SERVING
BI-Optimized
Aggregated, denormalized tables for Power BI / Tableau. Pre-computed KPIs. Row-level security applied. Materialized views for performance.
assembly_cost_summary part_cost_view supplier_performance revision_impact_view
Optimizations: Pre-aggregations RLS Security Partitioned by date Column masking

Key Design Decisions
Why File-Based for CAD?
CAD exports are periodic and large. File ingestion is simpler, more reliable for bulk, and avoids API overhead on large design files.
Why CDC for PLM?
Revision tracking is critical. CDC captures every INSERT/UPDATE/DELETE, enabling SCD Type 2 history — batch ingestion would miss changes.
Why Incremental for ERP (not CDC)?
High ERP transaction volume makes CDC expensive. Watermark-based incremental load is efficient and sufficient for cost data refresh frequency.
Why ADLS for all metadata?
Sources are read-only. Metadata stored in ADLS avoids SQL dependency, is cloud-native, scalable, and enables configuration-driven pipelines.
